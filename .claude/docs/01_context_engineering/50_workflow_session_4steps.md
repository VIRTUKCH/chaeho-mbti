# 50. 세션 워크플로우 — 탐색 → 계획 → 구현 → 커밋

> **내가 할 일:** AI에게 "X 만들어줘" 라고 한 번에 시키지 말고, **4단계로 분리해서** 시킨다. 짐코딩 강의의 시그니처 워크플로우.

---

## 1. 4단계 한 줄씩

| # | 단계 | AI가 하는 일 | 나의 역할 |
| --- | --- | --- | --- |
| 1 | **탐색** | 코드/문서를 읽고 현재 상태 파악 | 결과 확인 |
| 2 | **계획** | 무엇을 어떻게 할지 단계 나열 | 계획 검토·승인 |
| 3 | **구현** | 계획대로 코드 작성 | 진행 모니터링 |
| 4 | **커밋** | 메시지 작성 후 git commit | 메시지 확인 |

---

## 2. 왜 4단계로 쪼개나

### 한 번에 시키면 생기는 일

```
나: "MBTI 결과 페이지 만들어줘"
AI: (코드 짜기 시작) ... (수많은 가정과 추측) ... 완성!
나: 어... 이건 내가 원한 게 아닌데?
```

→ 결과물을 다 만들고 나서야 의도가 안 맞음을 깨닫는다. 토큰·시간 낭비.

### 4단계로 쪼개면

```
나: "MBTI 결과 페이지 만들고 싶어. 먼저 탐색."
AI: (현재 코드 구조 + MBTI 데이터 + 비슷한 페이지 분석) → 보고
나: 좋아. 그럼 계획.
AI: (1) src/app/result/[type]/page.tsx 만들기 (2) data fetch (3) 레이아웃 ...
나: (2) 빼고 (5) SEO 메타 추가해. 그대로 구현.
AI: (계획대로 구현)
나: 좋아. 커밋.
AI: ✨ feat: MBTI 결과 페이지 추가
```

→ **계획 검토 시점이 가장 비용 효율적인 교정 시점.**

---

## 3. 짐코딩이 이 워크플로우를 강조하는 이유

- 강의 헤드라인에 반복 등장하는 표어.
- 시청자 후기에서 가장 자주 인용되는 패턴.
- 인용: *"AI에게 무작정 구현시키지 말고 탐색·계획 단계를 분리하라는 사상."*

---

## 4. 각 단계에서 쓸 만한 명령

### 단계 1 — 탐색

```
"현재 src/app/ 구조를 보고, 비슷한 페이지가 있는지 확인해줘. 변경은 하지 마."
```

→ Read 만 쓰게 의도적으로 제한. Plan 모드를 켜두면 자동.

### 단계 2 — 계획

```
"위 분석을 바탕으로, X 를 구현할 단계별 계획만 적어줘. 코드 작성은 아직 하지 마."
```

→ 클로드 코드의 `EnterPlanMode` 와 `ExitPlanMode` 가 이 단계를 위한 도구.

### 단계 3 — 구현

```
"좋아. 위 계획 그대로 구현해. 다만 (n) 단계는 빼고."
```

### 단계 4 — 커밋

```
"/git:commit"  ← 슬래시 커맨드로 자동화
```

---

## 5. Plan 모드 활용

`.claude/settings.json` 에서 `defaultMode: "plan"` 설정해두면:
- 세션이 자동으로 Plan 모드로 시작
- AI 가 Edit/Write 를 못 함 → 자연스럽게 탐색·계획 단계가 됨
- 사용자가 `ExitPlanMode` 호출해야 구현 단계로 이동

→ 4단계 워크플로우를 settings 차원에서 강제하는 방법.

---

## 6. 짐코딩의 더 디테일한 버전: 프로젝트 단위 4-Phase

세션 4단계는 *한 작업* 단위. 프로젝트 *전체* 단위의 4-Phase 는 따로 있음:

```
Phase 1: 골격 (라우팅, 빈 페이지, 레이아웃)
Phase 2: UI (더미 데이터로 모든 화면)
Phase 3: 핵심 기능 (실제 데이터·로직)
Phase 4: 최적화 (성능·SEO·접근성)
```

→ 자세히: [51](./51_workflow_project_4phase.md)
